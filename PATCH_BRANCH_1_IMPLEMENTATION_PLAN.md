Закрыть быстрые доработки по ежедневному автотесту заявок без смешивания задач: один релиз = один пункт.

## Правило выполнения
1. Берём только один пункт из плана.
2. Делаем код + локальная проверка + CI-проверка.
3. Если пункт принят, только после этого переходим к следующему.
4. Никаких объединённых релизов с несколькими пунктами.

## Последовательность атомарных релизов

### R1. Формат алертов (единый шаблон)
- Объём: `send_step_alert`, `log_error`, `send_critical_alert`, итоговые summary.
- Результат: единый формат сообщений (время, лендинг, URL, шаг, ошибка, детали, ссылка на отчёт).
- Критерий приёмки: по каждому типу алерта формат совпадает с утверждённым шаблоном.
- Проверка: локальный запуск с искусственными падениями + проверка текста сформированных сообщений.

### R2. Автопопапы по таймеру как полноценные формы
- Объём: убрать логику “только закрыть”, включить автопоявляющиеся формы в полноценный цикл `fill -> submit -> confirm`.
- Результат: `profit`/аналогичные автоформы проходят те же проверки, что клик-формы.
- Критерий приёмки: автоформа попадает в отчёт как проверенная форма, а не как “закрытый оверлей”.
- Проверка: целевой домен с таймерным попапом + Allure шаги/вложения.

### R3. Обязательная отправка по вариантам услуг
- Объём: где есть варианты `В квартиру / Для бизнеса / В частный дом`, отправлять заявку по каждому варианту.
- Результат: один лендинг даёт отдельный результат по каждому варианту услуги.
- Критерий приёмки: падение одного варианта не скрывает результат остальных; в отчёте видно покрытие вариантов.
- Проверка: сайты с переключателями услуг, сверка количества фактических submit.

### R4. Проверка “закрытие Thanks и возврат на главную без региона”
- Объём: добавить отдельный шаг с ассертом после успешной отправки.
- Результат: фиксируется успешный возврат на главную и отсутствие блокирующего регион-попапа.
- Критерий приёмки: при неуспехе шага уходит понятный step-alert.
- Проверка: позитивный и негативный сценарии (искусственно сломанный возврат).

### R5. Переходы на внутренние страницы через навигацию
- Объём: заменить прямые `goto` на `/business` и городской `/business` на переходы через элементы навигации лендинга (с контролируемым fallback).
- Результат: сценарий соответствует ручному чек-листу “вход с главной + навигация по лендингу”.
- Критерий приёмки: прямой `goto` не является основным путём.
- Проверка: минимум 3 типа лендингов (с `/business`, без `/business`, с городским сценарием).

### R6. Мультибраузерность
- Объём: добавить запуск Suite A минимум в `chromium + firefox` (WebKit как расширение после стабилизации).
- Результат: CI и локальный запуск поддерживают выбор браузера/матрицу браузеров.
- Критерий приёмки: отчёт содержит результаты по браузерам, фатальных падений из-за запуска нет.
- Проверка: отдельный прогон по каждому браузеру на ограниченном наборе сайтов + полный ночной прогон.

### R7. Агрегированные алерты
- Объём: логика агрегации:
  - 1–5 страниц: обычные алерты;
  - >5 страниц одного лендинга: агрегированный алерт;
  - массовая ошибка по нескольким лендингам: сводный алерт;
  - алерт “исправлено” после восстановления.
- Результат: снижение спама и появление масштабных сводок.
- Критерий приёмки: при моделировании массовых падений отправляется агрегированный формат, а не поток одиночных сообщений.
- Проверка: тестовые наборы Allure results + dry-run отправки сообщений.

### R8. Сценарии adblock/antivirus (MVP)
- Объём: добавить поддерживаемый MVP-профиль запусков для проверки влияния блокировщиков; зафиксировать ограничения по антивирусным сценариям в CI.
- Результат: отдельный режим прогона “с блокировщиками” с прозрачной фиксацией статуса поддержки.
- Критерий приёмки: режим запуска документирован и воспроизводим, результаты попадают в Allure.
- Проверка: отдельный workflow/параметр запуска + сравнение результата с базовым прогоном.

## Что НЕ входит в эту ветку (отдельная patch-ветка)
- Проверка заявок в БД после отправки.
- Отдельный мобильный прогон форм Suite A.
- Внешний редактируемый источник URL/страниц (не через правку кода).
- Проверка соответствия улицы выбранному городу.
- Полноценный feature-flag “отключение алертов”.

## Минимальный тестовый гейт для каждого релиза
- Локально: точечный прогон затронутых сайтов (`--site=<domain>`).
- CI: успешный `Playwright Tests` + корректная генерация/publish Allure.
- Проверка Telegram-формата (на тестовом чате/в dry-run) для затронутых алертов.
