# План реализации: Patch Branch 3 (Provider Split + Orchestrated Runs)
Дата: 2026-04-27

## Цель ветки
Разделить текущий универсальный список сайтов на провайдерные конфиги и перейти к коротким последовательным прогонам по провайдерам с отдельной отчётностью.

Целевой эффект:
- быстрее локализовать падения;
- упростить редактирование конкретных лендингов;
- получить более короткие и понятные логи;
- сохранить текущую рабочую механику теста без регрессий.

## Бизнес-цели и критерии успеха
1. Любое изменение по сайту делается в конфиге провайдера, без правки тестовой логики.
2. Прогон можно запускать точечно по провайдеру и по отдельному сайту.
3. После завершения каждого провайдера формируется отдельный Allure и Telegram-итог.
4. Ошибка в одном провайдере не ломает запуск следующего.
5. Текущий full-run остаётся доступным как fallback-режим.

## Принципы реализации
1. Атомарность: один релиз = одна проверяемая цель.
2. Обратная совместимость: до полного перехода старый путь запуска остаётся рабочим.
3. Контракт конфигов: строгая схема + валидация до запуска браузера.
4. Наблюдаемость: каждый новый шаг оставляет понятные логи и диагностику.
5. Управляемый rollout: сначала ручной режим, затем ограниченный автозапуск.

## Что не входит в эту ветку
- Полная миграция расписания в Jenkins.
- Полноценная мобильная эмуляция в основном pipeline.
- Глубокая бизнес-валидация CRM/БД в универсальном тесте.

## Целевая структура (после PB3)
```text
config/
  providers/
    mts.py
    beeline.py
    megafon.py
    t2.py
    __init__.py
  loader.py
  schema.py
```

```text
.github/workflows/
  provider-orchestrator.yml
  provider-mts.yml
  provider-beeline.yml
  provider-megafon.yml
  provider-t2.yml
```

## Контракт провайдерного конфига (MVP)
```python
PROVIDER = "mts"
DEFAULT_CITY = "Москва"

SITES = [
    {
        "base_url": "https://mts-home-gpon.ru",
        "has_checkaddress": False,
        "has_business": True,
        "has_name_field": True,
        "cities": ["Москва"],  # опционально, если нет -> DEFAULT_CITY
    },
]
```

Правила:
- `base_url` обязателен;
- `cities` опционален, но должен быть списком строк;
- если `cities` не задан, используется `DEFAULT_CITY`;
- дубли `base_url` между провайдерами запрещены.

## Последовательность атомарных релизов

### P3-R0. Подготовка baseline и freeze интерфейсов
- Объём:
  - зафиксировать текущий рабочий запуск (`allure.yml`, `site_configs.py`, `test_universal2.py`);
  - зафиксировать тестовый набор сайтов для контрольного сравнения.
- Критерий приёмки:
  - есть baseline-результат до рефакторинга;
  - зафиксирован список smoke-сайтов для сравнения до/после.
- Проверка:
  - `core` прогон по 1 сайту каждого провайдера;
  - сохранённые артефакты Allure.
- Rollback:
  - откат к baseline commit без промежуточных конфликтов.

### P3-R1. Каркас провайдерных конфигов
- Объём:
  - создать директорию `config/providers`;
  - добавить пустые файлы провайдеров и `__init__.py`.
- Критерий приёмки:
  - импорт модулей проходит, запуск тестов не затронут.
- Проверка:
  - `python -m pytest --collect-only`.
- Rollback:
  - удалить новый каркас без изменений в рантайме.

### P3-R2. Вынос MTS в отдельный конфиг
- Объём:
  - перенести MTS-домены в `config/providers/mts.py`:
    - `mts-home-gpon.ru`
    - `mts-home.online`
    - `mts-home-online.ru`
    - `internet-mts-home.online`
    - `mts-internet.online`
- Критерий приёмки:
  - MTS сайты загружаются через новый конфиг;
  - фактическое поведение теста не изменилось.
- Проверка:
  - точечный `--provider=mts` (или временный загрузчик только MTS).
- Rollback:
  - возврат MTS записей в старый агрегированный источник.

### P3-R3. Вынос Beeline, Megafon, T2
- Объём:
  - аналогично P3-R2 для остальных провайдеров.
- Критерий приёмки:
  - все существующие сайты перенесены из старого списка в провайдерные файлы.
- Проверка:
  - `--collect-only` + точечные прогоны по одному сайту каждого провайдера.
- Rollback:
  - временный fallback на старый список.

### P3-R4. Лоадер совместимости
- Объём:
  - реализовать `config/loader.py`, который собирает провайдерные конфиги в прежний runtime-формат;
  - переключить `test_universal2.py` на новый loader.
- Критерий приёмки:
  - тестовый рантайм не знает о деталях разбиения и получает прежнюю структуру данных.
- Проверка:
  - сравнение списка сайтов до/после (count + set base_url).
- Rollback:
  - вернуть импорт старого `site_configs.py`.

### P3-R5. CLI-фильтры запуска (`provider`, `site`)
- Объём:
  - добавить параметр запуска по провайдеру;
  - сохранить совместимость с `--site`.
- Критерий приёмки:
  - `--provider=<name>` запускает только сайты провайдера;
  - `--site` продолжает работать как раньше.
- Проверка:
  - `pytest --collect-only` на комбинациях:
    - только `provider`;
    - только `site`;
    - `provider + site`.
- Rollback:
  - отключить provider-фильтр, оставив старую селекцию.

### P3-R6. Расширение городов через конфиг провайдера
- Объём:
  - поддержать в конфиге список `cities`;
  - для сценария города использовать первую валидную цель или `DEFAULT_CITY`.
- Критерий приёмки:
  - новый город добавляется только в конфиг, без правки теста;
  - Москва остаётся корректным дефолтом.
- Проверка:
  - точечный прогон сайта с альтернативным городом.
- Rollback:
  - временно игнорировать `cities`, использовать только `city_name`.

### P3-R7. Валидация конфигов
- Объём:
  - реализовать проверку схемы (`schema.py`);
  - запускать валидацию до выполнения тестов.
- Критерий приёмки:
  - ошибка конфига падает на старте с читаемым сообщением.
- Проверка:
  - негативные тесты на:
    - пустой `base_url`;
    - дубль домена;
    - неверный тип `cities`.
- Rollback:
  - отключить strict mode, оставить warning.

### P3-R8. Provider workflows (ручной режим)
- Объём:
  - добавить отдельные workflow по провайдерам;
  - каждый workflow запускает тесты только на своём конфиге.
- Критерий приёмки:
  - ручной запуск провайдера даёт отдельный Allure + Telegram summary.
- Проверка:
  - по одному ручному запуску на каждый workflow.
- Rollback:
  - запуск только через старый единый workflow.

### P3-R9. Orchestrator workflow (последовательная цепочка)
- Объём:
  - добавить `provider-orchestrator.yml`, который запускает провайдеры по порядку;
  - обеспечить продолжение цепочки даже при падении отдельного провайдера.
- Критерий приёмки:
  - отчётность по каждому провайдеру отдельная;
  - есть финальный общий summary.
- Проверка:
  - тестовый прогон с искусственным падением в середине цепочки.
- Rollback:
  - временно запускать провайдеры вручную без оркестратора.

### P3-R10. Telegram-логика для провайдерных прогонов
- Объём:
  - отправлять ошибки по провайдеру;
  - успешные одиночные шаги не спамить;
  - финальный сводный алерт по всей цепочке.
- Критерий приёмки:
  - меньше шума, больше полезной диагностики.
- Проверка:
  - dry-run + реальный прогон с 1 ошибочным провайдером.
- Rollback:
  - вернуть текущую схему алертов.

### P3-R11. README и пользовательская инструкция
- Объём:
  - детально обновить `README.md`:
    - структура конфигов;
    - как добавить сайт;
    - как добавить город;
    - как запускать провайдерные прогоны;
    - как читать отчёты и алерты.
- Критерий приёмки:
  - новый пользователь может добавить сайт/город и запустить прогон без чтения кода.
- Проверка:
  - smoke-чек по инструкции “с нуля”.
- Rollback:
  - оставить старые команды как legacy-раздел.

### P3-R12. Финальная приёмка и freeze
- Объём:
  - пройти контрольный список перед релизом.
- Критерий приёмки:
  - паритет с baseline на контрольной выборке;
  - стабильные точечные прогоны по провайдерам;
  - документация актуальна.
- Проверка:
  - 3 последовательных дня прогона без критичных регрессий.
- Rollback:
  - переключение на legacy full-run.

## Минимальный тестовый гейт для каждого релиза
1. `python -m pytest --collect-only` без ошибок.
2. Точечный smoke на затронутом провайдере.
3. Проверка корректности Telegram-логики (если затронута).
4. Проверка генерации Allure-артефакта.

## Риски и меры
1. Риск: “разъезд” форматов конфигов.
Мера: строгий schema-check + unit test на loader.
2. Риск: увеличение сложности workflow.
Мера: один orchestrator + единый шаблон provider-job.
3. Риск: потеря обратной совместимости.
Мера: сохранить legacy full-run до завершения P3-R12.
4. Риск: скрытые флейки при коротких прогонах.
Мера: 3-дневный stability soak перед фиксацией релиза.

## Задел под Jenkins и mobile-эмуляцию (future-ready)
1. Jenkins:
  - cron `08:00 MSK`;
  - trigger `workflow_dispatch` для orchestrator;
  - контроль ретраев и SLA.
2. Mobile эмуляция:
  - отдельный профиль исполнения;
  - reuse того же provider-loader и фильтров;
  - отдельный workflow и отчётность без смешивания с desktop.
3. Безопасное редактирование конфигов:
  - schema validation в CI;
  - pre-commit check на дубли доменов и обязательные поля.

## Формат отчётности по выполнению
После каждого атомарного релиза фиксируем:
- ID шага (`P3-Rx`);
- commit hash;
- что сделано;
- что проверено;
- остаточный риск/следующий шаг.
